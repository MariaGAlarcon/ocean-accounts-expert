# Framework & Foundational Guidance

This folder contains foundational documents for SEEA EA ecosystem accounting implementation.

## Contents

### `tiered_assessment_framework.md`
**Purpose:** Defines the tiered accuracy and feasibility assessment system used across all SEEA EA accounts.

**Use when:**
- Evaluating data quality and methodology rigor for your project
- Deciding between Tier 1, 2, or 3 approaches for different sub-procedures
- Justifying accuracy limitations in your analysis
- Planning pathway to improve data collection and methods over time

**Key sections:**
- Tier definitions (Feasibility vs. Accuracy)
- Assessment matrix for all ecosystem accounting components
- Binding constraint analysis
- Trade-off documentation

---

### `skill_data_quality_assessment.md`
**Purpose:** Systematic approach to assessing data quality across the accounting pipeline.

**Use when:**
- Planning your QA/QC procedures for raw field/remote-sensing data
- Documenting data issues found during analysis
- Deciding which data are fit for purpose vs. requiring additional processing
- Creating audit trails for regulatory or certification requirements

**Key sections:**
- Data quality dimensions (accuracy, completeness, consistency, timeliness)
- Quality assessment checklist
- Common data issues in coastal/marine ecosystem accounting
- Remediation strategies and trade-off documentation

## How These Documents Support Your Project

### Tiered Assessment Framework
Before starting a project:
1. Read this framework to understand the three tiers
2. Identify which sub-procedures matter most for your ecosystem
3. Assess current data/methods against tier definitions
4. Document your tier choice and justify why (e.g., "We use Tier 2 for fish biomass because...")
5. Plan pathway to upgrade if budget/capacity improves

### Data Quality Assessment
During project setup and throughout implementation:
1. Define quality standards for each data source (satellite, field survey, market data)
2. Apply the assessment checklist as data arrive
3. Document quality issues discovered during QA/QC
4. Make explicit decisions: fix, document as limitation, or exclude
5. Create audit trail showing data governance decisions
